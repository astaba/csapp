* The global =environ=
The reason we have both is primarily a matter of *scope and
flexibility*.  While =envp= is a convenient "gift" handed directly to
=main=, it suffers from a major limitation: it is local to that
function.

Here is why =environ= is necessary for a functional C system:

--------------

** 1. Accessibility Outside of =main=
:PROPERTIES:
:CUSTOM_ID: accessibility-outside-of-main
:END:
In a large-scale C project, many library functions (like =getenv=,
=setenv=, or =system=) need to access or modify environment variables.

- If only =envp= existed, you would have to manually pass that pointer
  through every single function call across your entire codebase just in
  case a low-level utility needs it.
- By using the global =environ= (the =.data= segment pointer), any
  function in any file can access the environment at any time without
  changing function signatures.

** 2. Dynamic Modification (The "Moving Target")
:PROPERTIES:
:CUSTOM_ID: dynamic-modification-the-moving-target
:END:
The environment is not static.  If you call =setenv()= or =putenv()=, the
environment might grow.

- *The Problem:* The original =envp= passed to =main= points to a fixed
  location on the *stack* (set up by the kernel).  The stack has limited
  space.
- *The Solution:* When you add enough new variables, =libc= will
  allocate a new, larger block of memory on the *heap*, copy the old
  variables there, and update the global =environ= pointer to point to
  this new heap location.
- *The Result:* The =envp= in your =main= arguments will still point to
  the /old, original/ stack location, while =environ= points to the
  /new, updated/ truth.

--------------

** The "Passing of the Torch" Logic
:PROPERTIES:
:CUSTOM_ID: the-passing-of-the-torch-logic
:END:
| Feature       | ~envp~ (in ~main~)             | ~environ~ (global)                     |
|               |                                | <20>                                   |
|---------------+--------------------------------+----------------------------------------|
| *Location*    | cOn the Stack (as an argument) | In the ~.data~ segment                 |
| *Scope*       | Local to ~main()~              | Global (Accessible everywhere)         |
| *Reliability* | Only reliable at program start | Updated by ~setenv~ / ~putenv~         |
| *Origin*      | Pushed by Kernel onto Stack    | Initialized by ~_start~ from the Stack |

--------------

** Historical Context: The "Third Argument"
:PROPERTIES:
:CUSTOM_ID: historical-context-the-third-argument
:END:
Historically, =envp= wasn't even part of the ISO C standard; it's a
POSIX extension.  The standard C signature for main is just
=int main(int argc, char **argv)=.  Without =environ=, a strictly
standard-compliant C program would have no way to reach the environment
at all!

*Warning:* If you are writing a program that modifies environment
variables and then checks them later, **always use =environ= or
=getenv()**=.  Relying on =envp= after a =setenv()= call can lead to
stale data or crashes.

***** What is happening under the hood?
:PROPERTIES:
:CUSTOM_ID: what-is-happening-under-the-hood
:END:
1. *The Initialization:* At process start, =_start= points the global
   =environ= variable to the exact same memory address on the *Stack*
   where the Kernel placed the environment pointers.
2. *The Threshold:* When you call =setenv=, =libc= realizes the
   Kernel-provided stack space is fixed in size.  It cannot simply "push"
   more data there without crashing into your program's actual data.
3. *The Relocation:* =libc= calls =realloc= or =malloc=.  It copies all
   existing environment pointers to a new block in the *Heap*, adds your
   new variable, and updates the =environ= pointer in the =.data=
   segment to this new address.
4. *The "Stale" Pointer:* The =envp= variable inside =main= is just a
   local pointer on =main='s stack frame.  It has no idea =libc= just
   moved the "source of truth." It still points to the old, original
   stack location.

***** Summary of Why This Matters
:PROPERTIES:
:CUSTOM_ID: summary-of-why-this-matters
:END:
- *Library Developers:* If you write a library, you *must* use =environ=
  because you don't have access to =main='s arguments.
- *Security:* Sometimes attackers try to manipulate the stack.  Knowing
  that =environ= can move to the heap is a key part of understanding
  process memory safety.
- *Consistency:* Always treat =envp= as a "read-only snapshot" of how
  the program started.  Treat =environ= as the "current state."

** 4.6 Usign =fork= and =execve= to Run Programs
*** Using =exit= in child process

Your "bad feeling" is actually a very sharp observation.  You are
touching on a subtle but important distinction in Unix systems
programming regarding how processes terminate.

In the context of the CS:APP shell example, let's break down why
=exit(0)= vs =_exit(0)= matters on line 44.

**** The Short Answer: Why you are right
:PROPERTIES:
:CUSTOM_ID: the-short-answer-why-you-are-right
:END:
In a child process that has failed to =execve=, using *=_exit()=* is
generally safer and more "correct" than =exit()=.

- *=exit()=* is a high-level library call.  It flushes standard I/O
  buffers and runs functions registered with =atexit()=.
- *=_exit()=* is a direct system call.  It terminates the process
  immediately without any user-space cleanup.

**** The Problem with =exit()= in a Forked Child
:PROPERTIES:
:CUSTOM_ID: the-problem-with-exit-in-a-forked-child
:END:
When you =Fork()= (line 41), the child gets a *copy* of the parent's
memory, including the state of the I/O buffers (like =stdout=).

1. If the parent has data sitting in a buffer that hasn't been printed
   yet, that data is now in the child's buffer too.
2. If the child calls =exit()=, it will flush those buffers.
3. Later, when the parent eventually exits or flushes, it will flush the
   same data.
4. *Result:* You might see duplicate output in your terminal.

--------------

**** Why CS:APP uses =exit(0)= anyway
:PROPERTIES:
:CUSTOM_ID: why-csapp-uses-exit0-anyway
:END:
While =_exit()= is technically "cleaner" for child processes that fail
to exec, the authors of CS:APP often use =exit()= for a few reasons:

1. *Simplicity:* For students, =exit()= is familiar.  Introducing
   =_exit()= requires explaining the difference between the C library
   and the system call interface prematurely.
2. *The =execve= behavior:* Since =execve= (line 42) wipes the process
   memory anyway if it /succeeds/, the only time line 44 is reached is
   if the command fails.  In this simple shell, there usually isn't
   enough buffered I/O to cause the "duplicate output" bug.
3. *Standardization:* The wrapper =unix_error= and other CS:APP wrappers
   often use standard =exit=.

**** Logic of the Process Flow
:PROPERTIES:
:CUSTOM_ID: logic-of-the-process-flow
:END:
The flow of this specific code is:

- *Line 41:* Process splits.
- *Line 42:* Child tries to transform into a new program.
- *Success:* The child's memory (and I/O buffers) is destroyed and
  replaced.
- *Failure:* The child continues to line 44.

#+begin_quote
*Pro Tip:* If you ever write a production-grade shell or a daemon,
definitely use =_exit()= in the child path after a failed =exec=.  It
prevents "double-flushing" and avoids triggering cleanup handlers that
were meant only for the parent.

#+end_quote

--------------

**** One more "Bad Feeling" you should have: Line 51

:PROPERTIES:
:CUSTOM_ID: one-more-bad-feeling-you-should-have-line-51
:END:
While you're looking at termination, look at line 51.  The parent uses
=waitpid(pid, &status, 0)=.  This shell is very basic---it doesn't handle
*zombie processes* for background jobs (=bg=).  If you run 100 commands
with =&=, and the parent never "waits" for them, those 100 processes
will stay in the process table as zombies.

** 5.4 Blocking and Unblocking Signals

*** Figure 8.32
Temporarily blocking a signal from being received.
#+begin_src C
   1  sigset_t mask, prev_mask;
   2
   3  Sigemptyset(&mask);
   4  Sigaddset(&mask, SIGINT);
   5
   6  /* Block SIGINT and save previous blocked set */
   7  Sigprocmask(SIG_BLOCK, &mask, &prev_mask);
   8  // ... Code region that will not be interrupted by SIGINT
   9  /* Restore previous blocked set, unblocking SIGINT */
  10  Sigprocmask(SIG_SETMASK, &prev_mask, NULL);
  11
#+end_src

*** Ignoring vs Blocking Signals
There is a subtle but vital distinction in systems programming: the
difference between *Blocking* a signal and *Ignoring* a signal.

While they both result in the signal not interrupting your code /right
now/, they behave very differently regarding the "memory" of the
event.

**** 1. The Core Difference: Memory vs. Deletion
:PROPERTIES:
:CUSTOM_ID: the-core-difference-memory-vs.-deletion
:END:
- *Blocking (=sigprocmask=):* This is like putting a letter in a
  *mailbox*.  The kernel sees the signal arrive, realizes you are busy,
  and puts it in a "Pending" queue.  As soon as you unblock it, the
  signal is delivered, and your handler runs.
- *Ignoring (=SIG_IGN=):* This is like throwing the letter in the
  *shredder*.  The kernel sees the signal arrive, sees your preference
  is =SIG_IGN=, and immediately discards it.  It is gone forever.

**** 2. Why your =SIG_IGN= snippet isn't "Blocking"
:PROPERTIES:
:CUSTOM_ID: why-your-sig_ign-snippet-isnt-blocking
:END:
In your example:

#+begin_src C
signal(signum, SIG_IGN);
// critical code runs here
signal(signum, SIG_DFL);
#+end_src

If a signal arrives while you are running that "critical code," it is
*discarded*.  When you restore =SIG_DFL= (the default behavior) on the
next line, the program has no idea a signal ever attempted to
arrive.  It will /not/ trigger the default action retroactively.

**** 3. The CSAPP Context
:PROPERTIES:
:CUSTOM_ID: the-csapp-context
:END:
When CSAPP talks about "Blocking," they are specifically referring to
*Postponement*.

- *Implicit Blocking:* If you are already in a handler for =SIGINT=,
  the kernel blocks =SIGINT= so you don't get a recursive "inception"
  of handlers.  If a second =SIGINT= hits, it waits.  When your first
  handler finishes, the second one triggers immediately.
- *Explicit Blocking:* You use =sigprocmask= to protect a "Critical
  Section" of code (like updating a global linked list) where you want
  the signal to happen, but just *not yet*.

**** 4. Comparison Table
:PROPERTIES:
:CUSTOM_ID: comparison-table
:END:
| Feature    | Blocking (=sigprocmask=)             | Ignoring (=SIG_IGN=)              |
|            |                                      | <20>                              |
|------------+--------------------------------------+-----------------------------------|
| *State*    | Signal is "Pending"                  | Signal is "Discarded"             |
| *Delivery* | Delivered as soon as unblocked       | Never delivered                   |
| *Analogy*  | *On Hold*: "Please stay on the line" | *Busy Signal*: "Click... Bzzzzzz" |
| *Use Case* | Protecting critical data structures  | Disabling features you don't want |

**** Summary
:PROPERTIES:
:CUSTOM_ID: summary
:END:
Your =SIG_IGN= trick is a valid way to protect a section of code, but
it is *lossy*.  If a user hits =Ctrl+C= while you are ignoring it, the
program will never react to that specific button press.

If you use =sigprocmask= (Blocking), the program will "wait" until
your critical code finishes and then *immediately* terminate (or run
the handler) the moment you unblock.  In systems programming,
*Blocking* is almost always preferred for critical sections because it
doesn't "lose" user input or system events.

*** Even explicit blocking doesn't redeem the third signal
#+begin_quote
First signal is handled with.  While first is hndled: Second signal is
postponed through the bit vector; Third signal is disarded and lost
for good.  Is explicit blocking allow to retreive the third signal???
#+end_quote

The short answer is: *No.* Even with explicit blocking, the third signal
is lost forever.

Explicit blocking (=sigprocmask=) does not change the fundamental
architecture of the Linux signal system, which is *bit-vector based*,
not *queue-based*.

**** 1. The "Pending" Bit Vector (The Hardware Reality)
:PROPERTIES:
:CUSTOM_ID: the-pending-bit-vector-the-hardware-reality
:END:
Inside the Kernel, for every process, there is a *Pending Bit Vector*.
Imagine a 64-bit integer where each bit represents a signal type:
- Bit 2 is =SIGINT=.
- Bit 17 is =SIGCHLD=.

When a signal is "Pending" (because it is blocked), the Kernel simply
flips that bit from =0= to =1=.

- *First Signal:* The bit is =0=.  The Kernel flips it to =1= and tries
  to deliver it.
- *Second Signal:* If the first is still being handled (or is blocked),
  the Kernel looks at the bit.  It is already =1=.  *It stays =1=.*
- *Third Signal:* The Kernel looks at the bit.  It is already =1=.  *It
  stays =1=.*

There is no "counter" and no "list." The Kernel only knows *if* a signal
of that type is waiting, not *how many* times it was sent.

**** 2. Explicit Blocking vs. Implicit
:PROPERTIES:
:CUSTOM_ID: explicit-blocking-vs.-implicit
:END:
Explicit blocking (=sigprocmask=) simply tells the Kernel: "Check the
Pending bit vector, but don't act on it until I give you the green
light."

It doesn't provide a bigger "bucket" to catch signals; it just keeps the
"shutter" closed longer.  Whether you block for 1 second or 1 hour, that
bit can only be "on" or "off."

**** 3. The Only Exception: Real-Time Signals
:PROPERTIES:
:CUSTOM_ID: the-only-exception-real-time-signals
:END:
Standard signals (1--31, like =SIGINT=, =SIGKILL=) are non-queued.
However, Linux supports *Real-Time Signals* (typically =SIGRTMIN= to
=SIGRTMAX=).

If you use =sigqueue()= to send these and install handlers with the
=SA_SIGINFO= flag:
1. They *are* queued.
2. If you send 3, the process will eventually receive 3.
3. They can carry a payload (an integer or pointer).

**** 4. Why is it designed this way?
:PROPERTIES:
:CUSTOM_ID: why-is-it-designed-this-way
:END:
Standard signals are intended to represent *state changes*, not *data
streams*.  * *Analogy:* A signal is like a "Fire Alarm." * If one person
pulls the alarm, the bell rings.  * If ten people pull ten different
alarms simultaneously, the bell still just rings.  * You don't need the
bell to ring ten times to know the building is on fire; you just need to
know the /state/ has changed to "Fire."

**** Summary for your mental model:
:PROPERTIES:
:CUSTOM_ID: summary-for-your-mental-model
:END:
- *Explicit Blocking* allows you to control *when* you react.
- *Standard Signals* only allow you to know *if* an event happened since
  the last time you checked.
- The *Third Signal* is always "shredded" because there is no place in
  the bit-vector to store its existence once the "Pending" bit for that
  type is already set.

** 5.5 Writing Signal Handlers

*** Safe Signal Handling
*Our aim here is to give you some conservative guidelines for writing
handlers that are safe to run concurrently.  If you ignore these
guidelines, you run the risk of introducing some subtle concurrency
errors.  With such errors your program works correctly most of the
time.  However, when it fails, it fails in unpredictable and
unrepeatable ways the are horrendously difficult to debug.  Forewarned
is forearmed!*

**** G0: Keep handlers as simple as possible
:PROPERTIES:
:CUSTOM_ID: gO
:END:
For example, the handler might simply set a global flag and return
immediately.  While all processing associated with the receipt of the
signal is performed by the main program, which periodically checks
(and resets) the flag.

**** G1: Call only /async-signal-safe/ functions in handlers
:PROPERTIES:
:CUSTOM_ID: g1
:END:
*Async-signal-safe* functions can be safely called from handler for
because they have either of two properties:

1. *Reentrant-safe:* Never access varaibles out of its scope.
2. *Uninterruptible* by signal handlers.

Notice that many popular functions, such as =printf=, =sprintf=,
=malloc=, and =exit=, are not on the list *Async-signal-safe
functions* list ([[shell:man 7 signal-safety]]).

The only safe way to generate output from a signal handler is to use
the =write= function (see Section 10.1).  In particular, calling
=printf= or =sprintf= is unsafe.

References:
- [[file:~/coding/c_prog/csapp/src/csapp.c::/* Figure 8.34 The Sio (Safe I/O) package for signal handlers.][Figure 8.34 csapp]] The Sio (Safe I/O) package for signal handlers.
- [[file:sigintsafe.c][Figure 8.35 sigintsafe]] A safe version of the SIGINT handler from
  [[file:sigint.c][Figure 8.30 sigint]].

**** G2: Save and restore ~errno~
Since many /async-signal-safe/ functions set ~errno~ when they return on error, calling them inside a handler might interfere with other parts of the program that rely on errno.

- If the handler returns to the calling flow, save it in a variable at
  entry and restore it before returning.
- If the handler terminates all flow with =_exit()=, don't bother
  about ~errno~.

**** G3: Protect accesses to shared global data structures by blocking all signals
:PROPERTIES:
:CUSTOM_ID: protect-accesses-to-shared-global-data-structures
:END:
The sequence of instructions necessary to access a blobal data
structure ~d~ from the main proram could be interrupted by the
handler.  In that event, the handler might find ~d~ in a inconsistent
state, with unpredictable results.

- *Curtail those structures from signals* when accessing them both in main program and handlers.

**** G4: Declare global variable with ~volatile~
Consider a global variable ~g~ shared by a handler and the main
routine: handler updates ~g~ and main checks it periodically.

To an *optimizing compiler* it would appear value of ~g~ never changes
within main.  The reason is, a handler call as an
Exception-Control-Flow is an hypothetical event completely outside of
the compiler-predictable flow control based on *syntactic branching
logic.* Thus it would be safe to satisfy all references to ~g~ with a
copy cahed in a register.  Consequently, main would never see any
updated value of ~g~.  Declaring:

#+begin_src C
  volatile int g;
#+end_src

would force the compiler to read every reference to ~g~ from memory.

Of course *all access to global data shouls protected by temporarily
blocking all signals.*

**** G5: Declare /flags/ with ~sig_atomic_t~
In the suggested design at [[#gO][G0 Guideline]], *C* allows to assure atomic
access of flags shared in this way, by declaring them with a special
*integer data type* as:

#+begin_src C
volatile sig_atomic_t flag;
#+end_src

*Notice*: The atomicity only applies to *individual read and write*.  It won't work for multiple-instruction statements like:

#+begin_src C
  flag++;				/* or */
  flag = flag + 10;
#+end_src

**** SIGCHLD for production grade server
:PROPERTIES:
:CUSTOM_ID: sigchld-for-production-grade-server
:END:
The way [[file:signal2.c][Figure 8.37 signal2]] reaps children is the correct way, but the
third argument to ~waitpid()~ set to ~options: 0~ makes it block as
long as a running child is not dead to be reaped.  That blocking

handlers to return before it can service incoming connections.

***** Passing ~WNOHANG~ the bad way
Passing to ~waitpid()~ the third argument ~options: WNOHANG~ tells it
not to block but to return ~0~ if children are there but not ready for
harvest.  That sounds great but entails another fatal flaw.  The
returned ~0~ would break the ~while~ loop with children still to be
reaped.  The next ~if~ claused would then check for a *stale* ~errno~
of very bad omen.  With ~errno~ most likely different of ~ECHILD~ since
children are still running the ~Sio_error()~ would trigger and kill
the whole process.  Nasty outcome.

#+begin_src C
  /* FIXME: Bad code */
  while (waitpid(-1, NULL, WNOHANG) > 0) {
    Sio_puts("Handler reaped child\n");
  }
  if (errno != ECHILD)
    Sio_error("waitpid error");
#+end_src

***** Reaping children the right way
:PROPERTIES:
:CUSTOM_ID: reaping-children-the-right-way
:END:
Avoid stale values and *only ever check for ~errno~ when an error
actually occured.*

**** Signal-protected Statement Blocks are not Atomic
#+begin_quote
After reading [[#protect-accesses-to-shared-global-data-structures][G3 Guideline]] , I wonder if blocking signals around a
block of statement is equivalent to make them atomic, e.i., exonerated
from contex switches and garanteed to run in one shot?
#+end_quote

This is a crucial distinction.  The short answer is: *No.* Blocking
signals makes a block of code *atomic with respect to signal handlers*,
but it does *not* make it atomic with respect to the *CPU or the
Scheduler.*

To understand this, you have to separate "Interrupts" into two
categories: *Hardware/Signal Interrupts* and *Context Switches.*

***** 1. The Signal "Shield" (What it DOES do)
:PROPERTIES:
:CUSTOM_ID: the-signal-shield-what-it-does-do
:END:
When you block all signals, you are essentially telling the kernel: /"If
a signal arrives, do not jump into a handler function while I am inside
this block."/

From the perspective of your *Signal Handler*, the code is now "atomic."
The handler cannot "interrupt" your global data update halfway through.
This prevents the "Race Condition" where the parent is adding to a list
and the child-reaper is deleting from it at the same time.

***** 2. The Scheduler "Context Switch" (What it DOES NOT do)
:PROPERTIES:
:CUSTOM_ID: the-scheduler-context-switch-what-it-does-not-do
:END:
Blocking signals *does not stop the Linux Scheduler.* The Kernel can
still decide that your process's "time slice" is up.  It will:
1. Freeze your process right in the middle of your "protected" block
2. Save your registers.
3. Switch to a different process (like a web browser or a compiler).
4. Later, switch back to your process and resume exactly where it left off.

Since the other process lives in a different virtual memory space, it
cannot see your global data.  Therefore, this "interruption" is
*invisible* to your program's logic and doesn't cause a race condition.

***** 3. "Atomic" vs. "Safe"
:PROPERTIES:
:CUSTOM_ID: atomic-vs.-safe
:END:
In systems programming, we use different tools for different types of
"interruption":

| Threat                  | Prevention Tool    | Effect                                                                                             |
|                         |                    | <20>                                                                                                |
|-------------------------+--------------------+----------------------------------------------------------------------------------------------------|
| *Signal Handlers*       | =sigprocmask=      | Prevents handler from running on the same CPU core as your main thread.                            |
| *Context Switches*      | /N/A (User space)/ | You cannot usually prevent these in standard Linux user-space.                                     |
| *Multiple Threads/CPUs* | *Mutexes / Locks*  | Prevents two different threads on two different CPU cores from touching the data at the same time. |

***** 4. Why CSAPP says "Protect Accesses"
:PROPERTIES:
:CUSTOM_ID: why-csapp-says-protect-accesses
:END:
In the specific context of Figure 8.40, the "Race" is only between the
*Parent's =main()=* and the *Parent's =handler()=*.  Both of these run in
the same process memory.

The only way the =handler= can run is if a =SIGCHLD= arrives.  By
blocking =SIGCHLD=, you ensure that the =main()= function can finish
=addjob= before the =handler= is allowed to run and call =deletejob=.

***** The Emacs Analogy
:PROPERTIES:
:CUSTOM_ID: the-emacs-analogy
:END:
Think of it like working in *Emacs*.  * *Blocking Signals:* Is like
turning off =auto-save-mode= or specialized hooks while you perform a
complex regex replace.  You are preventing /Emacs itself/ from triggering
internal functions that might mess up your buffer while you are
mid-edit.  * *Context Switching:* Is like you minimizing the Emacs window
to check a browser.  Emacs is "frozen," but when you click back into it,
the buffer is exactly as you left it.  The "interruption" (minimizing)
didn't change the /internal state/ of your text.

***** Summary
:PROPERTIES:
:CUSTOM_ID: summary
:END:
Blocking signals makes a block of code *"Signal-Safe"* and
*"Reentrant-Safe"*, but the kernel is still free to context-switch you
out at any time.  Because context switches are transparent to the process
state, we generally don't care about them for data integrity---we only
care about the asynchronous "jumps" into handler code.

If you were doing *Kernel* programming, you would "disable interrupts"
to achieve true, one-shot hardware atomicity, but in *User* programming
(CSAPP), =sigprocmask= is the closest shield we have!

I understand the confusion.  The "magic" isn't in what the =sigemptyset=
function does, but in how the *Kernel* interprets the =struct sigaction=
once you hand it over.

To understand Line 6, you have to separate two different "Blocking"
behaviors that happen when a signal handler is running.

**** 1. The Implicit Block (Automatic)
:PROPERTIES:
:CUSTOM_ID: the-implicit-block-automatic
:END:
The comment in the CSAPP snippet is slightly misleading if read in
isolation.  By default, POSIX =sigaction= *always* blocks the signal of
the type currently being handled.  * If you are in a =SIGINT= handler,
the kernel automatically adds =SIGINT= to the signal mask for the
duration of that handler.  * You don't have to do anything to make this
happen; it's a safety feature to prevent a signal from interrupting its
own handler recursively.

**** 2. The =sa_mask= (The "Extra" Block)
:PROPERTIES:
:CUSTOM_ID: the-sa_mask-the-extra-block
:END:
The =sa_mask= field in the =struct sigaction= is for *additional*
signals you want to block while the handler is running.  * If you are
handling =SIGINT=, but you also want to make sure =SIGQUIT= doesn't
interrupt you, you would add =SIGQUIT= to =action.sa_mask=.

**** 3. Why =sigemptyset=?
:PROPERTIES:
:CUSTOM_ID: why-sigemptyset
:END:
Line 6 (=sigemptyset(&action.sa_mask)=) is there to ensure that *no
other* signals are blocked except for the one currently being handled.

The code is essentially saying:
1. "Set the handler to my function."
2. *"Clear the extra mask."* (I don't want to block any /additional/
   signals while this handler runs).
3. "Let the kernel handle the default implicit blocking of the current
   signal."

**** 4. Why is it necessary to empty it?
:PROPERTIES:
:CUSTOM_ID: why-is-it-necessary-to-call-it
:END:
In C, local variables on the stack (like =struct sigaction action=) are
*not initialized to zero*.  They contain "garbage" data from whatever was
previously at that memory location.
- If you don't call =sigemptyset=, =action.sa_mask= might contain random bits that tell the kernel to block every single signal on the system!
- Line 6 is "initializing" the mask to a clean state (all zeros).

**** Summary
:PROPERTIES:
:CUSTOM_ID: summary
:END:
The comment "Block sigs of type being handled" on Line 6 is actually
describing the *result* of using =sigaction= in general, combined with
the fact that we aren't adding any /extra/ signals to the mask.

*The logic is:*
1. Initialize (empty) the mask so we don't block random signals.
2. Rely on the Kernel's default behavior to block the current signal
   type while the handler executes.

** 5.6 Synchronizing Flows to Avoid Nasty Concurrency Bugs
References:
- [[file:~/coding/c_prog/csapp/ecf/procmask1.c][Figure 8.39 procmask1]] *A shell program with a subtle synchronization
  error.* If the child terminates before the parent is able to run,
  then ~addjob~ and ~deletejob~ will be called in the wrong order.
- [[file:procmask2.c][Figure 8.4O procmask2]] *Using* sigprocmask *to synchronize
  processes.* In this example, the parent ensures that ~addjob~
  executes before the corresponding ~deletejob~.

The name =prev_one= is indeed a bit of a "shorthand" that O'Hallaron and
Bryant use, and it can be misleading.  To understand it, you have to look
at *exactly what was in the mask* right before that specific call.

*** 1. The "State" vs. The "Change"
:PROPERTIES:
:CUSTOM_ID: the-state-vs.-the-change
:END:
The third argument of =sigprocmask= doesn't just hold "the old signals
that were blocked"; it holds the *entire signal mask* (the bit-vector)
as it existed the moment before the function was called.

In this specific code:

1. *Initial State:* Most signals are unblocked (the mask is mostly
   =0=s).
2. *=Sigprocmask(SIG_BLOCK, &mask_one, &prev_one)=*:
   - You are telling the kernel: "Add =SIGCHLD= to the current mask."
   - The kernel looks at the current mask (which is empty/default) and
     saves it into =prev_one=.
   - *=prev_one= now contains the "mostly empty" mask.*

*** 2. Why the name =prev_one=?
:PROPERTIES:
:CUSTOM_ID: why-the-name-prev_one
:END:
The authors use the name =prev_one= to mean *"The PREVious mask that
existed before I blocked ONE signal (SIGCHLD)."*

It is a "Restore Point." They aren't naming it based on what is /in/ it,
but based on /why/ they saved it.  They want to be able to go back to the
state where =SIGCHLD= was not blocked.

*** 3. The "Unblocking" Logic
:PROPERTIES:
:CUSTOM_ID: the-unblocking-logic
:END:
Look at the end of the =while= loop in =main=:

#+begin_src C
Sigprocmask(SIG_SETMASK, &prev_one, NULL);
#+end_src

By using =SIG_SETMASK=, they are saying: "I don't care what the mask is
right now; throw it away and replace it with exactly what was in
=prev_one=." Since =prev_one= was saved back when =SIGCHLD= was
unblocked, this line *unblocks* =SIGCHLD=.

*** 4. The =mask_all= / =prev_all= contrast
:PROPERTIES:
:CUSTOM_ID: the-mask_all-prev_all-contrast
:END:
In the =handler=, they do this:

#+begin_src C
Sigprocmask(SIG_BLOCK, &mask_all, &prev_all);
deletejob(pid);
Sigprocmask(SIG_SETMASK, &prev_all, NULL);
#+end_src

Here, =prev_all= saves the mask that existed *before they blocked ALL
signals*.  They do this to ensure =deletejob= (which modifies a global
list) is completely atomic and cannot be interrupted by /anything/.

*** 5. Common Confusion: Does it "stack"?
:PROPERTIES:
:CUSTOM_ID: common-confusion-does-it-stack
:END:
It is important to realize that =sigprocmask= doesn't have a "memory" or
a "stack" of its own.

- If you call =SIG_BLOCK= twice, you don't "unblock" twice to get back
  to the start.
- You are responsible for saving the "Old State" in a variable (like
  =prev_one=) so you can manually "Set" the mask back to that state
  later.

*** Summary
:PROPERTIES:
:CUSTOM_ID: summary
:END:
Think of it like this:

- *=mask_one=*: The "Request" (Please block =SIGCHLD=).
- *=prev_one=*: The "Snapshot" (What did the mask look like before I
  made that request?).

*Regarding your Emacs note:* Since you like your environment to persist
(like =winner-mode=), think of =prev_one= as a *Window Configuration* in
Emacs.  Before you perform a disruptive action (like splitting the screen
into 10 parts), you save the "previous" configuration so you can restore
it exactly as it was with a single command.

In C, =prev_one= is your "Undo" buffer for signal settings!

** 5.7 Explicitly Wait for Signals
References:
- [[file:waitforsignal.c][Figure 8.41 waitforsignal1]] Waiting for a signal with a spin
  loop.  This code is correct, but the spin loop is wasteful.
- [[file:~/coding/c_prog/csapp/ecf/sigsuspend.c][Figure 8.42 sigsuspend]] Waiting for a signal with sigsuspend.

#+begin_quote
What is the point of catching SIGINT in a way that disables it? It
makes the program difficult to kill.
#+end_quote

You've spotted a classic "textbook detail" that can be incredibly
annoying in practice.  Why would O'Hallaron and Bryant include an empty
=SIGINT= handler in a program that is supposed to demonstrate signal
synchronization?

The reason is actually about *demonstrating how =sigsuspend= (or in this
case, a spin loop) reacts to /any/ signal.*

*** 1. Preventing Program Death during the "Wait"
:PROPERTIES:
:CUSTOM_ID: preventing-program-death-during-the-wait
:END:
In this specific program, the Parent is in a tight loop:

1. Fork a child.
2. Unblock =SIGCHLD=.
3. *Spin until =pid= is non-zero.*

If you didn't have the =SIGINT= handler, hitting =Ctrl+C= would invoke
the *Default Action*, which is to terminate the process immediately.
By installing an empty handler, the authors ensure that if you hit
=Ctrl+C=, the following happens:

- The kernel jumps to the empty =sigint_handler=.
- It returns immediately.
- The Parent *remains alive* and continues spinning in the =while(!pid)=
  loop.

*** 2. The Theoretical Point (Signal Interruption)
:PROPERTIES:
:CUSTOM_ID: the-theoretical-point-signal-interruption
:END:
The authors likely included it to show that the "Wait" (the spin loop)
can be interrupted by signals /other/ than the one you are waiting for.

In a real shell (which this chapter is teaching you to build), you often
want the shell to stay alive even if the user hits =Ctrl+C= to kill the
/foreground job/.  If the shell didn't "catch" (and effectively ignore)
=SIGINT=, the shell itself would die every time the user tried to stop a
running program.

*** 3. Why it's a "Bad" UX here
:PROPERTIES:
:CUSTOM_ID: why-its-a-bad-ux-here
:END:
As you noted, it makes the program a "zombie" that is hard to kill.
Since the loop is:

#+begin_src C
while (1) {
    // ... fork ...
    while (!pid) ;
}
#+end_src

If you hit =Ctrl+C=, you trigger the empty handler, the loop continues,
and you are still stuck.

*How to kill it:* Because you only "caught" =SIGINT=, you can still kill
it using:

- =Ctrl+\= (SIGQUIT - usually mapped to the backslash).
- =kill -9 <pid>= from another terminal.

*** 4. The "Emacs" Perspective
:PROPERTIES:
:CUSTOM_ID: the-emacs-perspective
:END:
Think of this empty handler like a *hook that does nothing but
=ignore=*.

In Emacs, if you have a keybinding that is dangerous, you might bind it
to a function that just returns =nil= so that the default behavior
(which might close your buffer or delete your =winner-mode= history) is
suppressed.

Here, the =SIGINT= handler is a "noop" hook designed to prevent the
"Default OS Hook" (Termination) from running.

*** Summary
:PROPERTIES:
:CUSTOM_ID: summary
:END:
The =SIGINT= handler is there to show that the process *stays in the
loop* regardless of other system events.  However, in later sections
(like Figure 8.42), they replace this wasteful "spin loop" with
=sigsuspend=.  The empty =SIGINT= handler is kept there to prove that
=sigsuspend= will wake up, see that =pid= is still =0= (because it
wasn't a =SIGCHLD=), and go back to sleep---which is a much more
efficient way to wait.

It's a pedagogical choice to show *robustness*, even if it makes the
test program feel like a trap!

** Problem 8.22
:PROPERTIES:
:CUSTOM_ID: problem-8-22
:END:
Write your own version of the Unix =system= function as:

#+BEGIN_SRC C
  int mysystem(char *command);
#+end_src

The =mysystem= function executes command by invoking =/bin/sh -c
command=, and then returns after =command= has completed.  If =command=
exits normally (by calling the =exit= function or executing a =return=
statement), then mysystem returns the command exit status.  For
example, if command terminates by calling =exit(8)=, then =mysystem=
returns the value =8=.  Otherwise, if =command= terminates abnormally,
then =mysystem= returns the status returned by the shell.

*** The purpose of =system= library function
Let's place =system()= in the context of the classic *process‑spawning
chain* (=fork=, =exec*=, =wait*=).

**** What =system()= Does
- =system(const char *command)= is a *library function* in C (from
  =<stdlib.h>=).
- It takes a shell command string (e.g., ="ls -l /tmp"=) and runs it as
  if you typed it in your shell.
- Internally, it does the following:
  1. *Forks* a child process.
  2. In the child, it *execs* the user's shell (usually =/bin/sh=) with
     the command string.
  3. The parent process *waits* for the child to finish, then returns
     the child's exit status.

So =system()= is essentially a convenience wrapper around the =fork= →
=exec= → =wait= chain.

**** Comparison
| Aspect                 | Manual ~fork~ + ~exec~ + ~wait~          | ~system()~                                                           |
|                        | <20>                                     | <20>                                                                 |
|------------------------+------------------------------------------+----------------------------------------------------------------------|
| Control over arguments | Full control (you pass argv, envp)       | Limited --- you pass a single command string                         |
| Shell involvement      | Not required (execs program directly)    | Always runs through a shell (~/bin/sh -c~)                           |
| Error handling         | You handle ~fork~/~exec~ errors yourself | ~system()~ returns ~-1~ on fork error, or child's exit status        |
| Convenience            | Verbose, but precise                     | Very easy, one call                                                  |
| Security               | Safer (no shell parsing)                 | Risky if command string comes from untrusted input (shell injection) |

**** Example
Manual approach:

#+BEGIN_SRC C
  pid_t pid = fork();
  if (pid == 0) {
      execl("/bin/ls", "ls", "-l", "/tmp", (char *)NULL);
      perror("execl failed");
      exit(EXIT_FAILURE);
  } else {
      int status;
      waitpid(pid, &status, 0);
  }
#+END_SRC

Using =system()=:

#+BEGIN_SRC C
  system("ls -l /tmp");
#+END_SRC

**** Purpose in the Chain
- =system()= is a *high‑level shortcut*: it hides the details of =fork=,
  =exec=, and =wait=.
- It's great for quick commands, scripts, or prototyping.
- But if you need fine‑grained control over arguments, environment, or

** Problem 8.23
:PROPERTIES:
:CUSTOM_ID: problem-8-23
:END:
One of your colleagues is thinking of using signals to allow a parent
process to count events that occur in a child process.  The idea is to
notify the parent each time an event occurs by sending it a signal and
letting the parent’s signal handler increment a global counter
variable, which the parent can then inspect after the child has
terminated.  However, when he runs the test program in [[file:~/coding/c_prog/csapp/ecf/counterprob.c][Figure 8.45]] on
his system, he discovers that when the parent calls =printf=, counter
always has a value of 2, even though the child has sent five signals
to the parent.  Perplexed, he comes to you for help.  Can you explain
the bug?

*** Answer
Although 5 signals are sent from child to parent, at parent's exit
src_C{counter = 2;}.

The raison is signals are not queued.  After installing a =s_handler=
to catch =s= signal, a first =s= signal is caught and handles by
=s_handler=.  If a second =s= signal arrives while =s_handler= is
running, the second signal is added to the pending vector by the
system for later handling.  Any subsequent arrival of =s= type signal
while the handler is running are not queued but outright discarded and
lost for good.  When the handler returns the system sees in the
pending vector the "ON" bit for the =s= signal and forces the process
to handle the pending signal setting =counter= to 2.
