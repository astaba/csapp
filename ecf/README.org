* 8.4.5 The global =environ=

The reason we have both is primarily a matter of *scope and
flexibility*. While =envp= is a convenient "gift" handed directly to
=main=, it suffers from a major limitation: it is local to that
function.

Here is why =environ= is necessary for a functional C system:

--------------

** 1. Accessibility Outside of =main=
:PROPERTIES:
:CUSTOM_ID: accessibility-outside-of-main
:END:
In a large-scale C project, many library functions (like =getenv=,
=setenv=, or =system=) need to access or modify environment variables.

- If only =envp= existed, you would have to manually pass that pointer
  through every single function call across your entire codebase just in
  case a low-level utility needs it.
- By using the global =environ= (the =.data= segment pointer), any
  function in any file can access the environment at any time without
  changing function signatures.

** 2. Dynamic Modification (The "Moving Target")
:PROPERTIES:
:CUSTOM_ID: dynamic-modification-the-moving-target
:END:
The environment is not static. If you call =setenv()= or =putenv()=, the
environment might grow.

- *The Problem:* The original =envp= passed to =main= points to a fixed
  location on the *stack* (set up by the kernel). The stack has limited
  space.
- *The Solution:* When you add enough new variables, =libc= will
  allocate a new, larger block of memory on the *heap*, copy the old
  variables there, and update the global =environ= pointer to point to
  this new heap location.
- *The Result:* The =envp= in your =main= arguments will still point to
  the /old, original/ stack location, while =environ= points to the
  /new, updated/ truth.

--------------

** The "Passing of the Torch" Logic
:PROPERTIES:
:CUSTOM_ID: the-passing-of-the-torch-logic
:END:
| Feature       | =envp= (in =main=)             | =environ= (global)                     |
|---------------+--------------------------------+----------------------------------------|
| *Location*    | On the Stack (as an argument)  | In the =.data= segment                 |
| *Scope*       | Local to =main()=              | Global (Accessible everywhere)         |
| *Reliability* | Only reliable at program start | Updated by =setenv= / =putenv=         |
| *Origin*      | Pushed by Kernel onto Stack    | Initialized by =_start= from the Stack |

--------------

** Historical Context: The "Third Argument"
:PROPERTIES:
:CUSTOM_ID: historical-context-the-third-argument
:END:
Historically, =envp= wasn't even part of the ISO C standard; it's a
POSIX extension. The standard C signature for main is just
=int main(int argc, char **argv)=. Without =environ=, a strictly
standard-compliant C program would have no way to reach the environment
at all!

#+begin_quote
*Warning:* If you are writing a program that modifies environment
variables and then checks them later, **always use =environ= or
=getenv()**=. Relying on =envp= after a =setenv()= call can lead to
stale data or crashes.

#+end_quote

* 8.4.5 =environ= migration

Here is a demonstration of the "migration." This code will prove that
while =envp= is a snapshot of the birth of the process, =environ= is the
living, breathing record of the process's state.

** The Experiment: Stack vs. Heap
:PROPERTIES:
:CUSTOM_ID: the-experiment-stack-vs.-heap
:END:
This script compares the addresses before and after forcing the
environment to grow.

#+begin_src C :tangle ymyecho.c
  /* Created on: Thu Feb 12 15:13:47 +01 2026 */
  /* Practice Problem 8.6 (solution page 833): Write a program called
  myecho that prints its command-line arguments and environment
  variables. This program shows the difference between the golbal
  'environ' form the .data segment and the 'envp' as an argument of the
  'main()' function. */

  #include <stdio.h>
  #include <stdlib.h>

  extern char **environ;

  int main(int argc, char *argv[], char *envp[]) {
    size_t i;
    printf("--- INITIAL STATE ---\n");
    printf("envp    (stack): %p\n", (void *)envp);
    printf("environ (.data): %p\n", (void *)environ);

    /* 1. Check if they point to the same array initially */
    if (envp == environ) {
      printf("Result: 'envp' and 'environ' point to the same stack address.\n");
    }
    /* 2. force the environment to grow significantly. This will exceed the
     ,* original stack allocation for the env array. */
    printf("\n--- MODIFYING ENVIRONMENT ---\n");
    for (i = 0; i < 100; i++) {
      char name[10], value[10];
      sprintf(name, "NAME_%03d", (int)i);
      sprintf(value, "VALUE_%03d", (int)i);
      setenv(name, value, 1);
    }

    printf("Original envp (stack): %p\n", (void *)envp);
    printf("New environ   (.data): %p\n", (void *)environ);

    if (envp != environ) {
      printf("\nResult: 'environ' has migrated to the HEAP!\n");
      printf("The original 'envp' argument is now STALE/OUTDATED.\n");
    }

    return EXIT_SUCCESS;
  }
#+END_SRC

--------------

** What is happening under the hood?
:PROPERTIES:
:CUSTOM_ID: what-is-happening-under-the-hood
:END:
1. *The Initialization:* At process start, =_start= points the global
   =environ= variable to the exact same memory address on the *Stack*
   where the Kernel placed the environment pointers.
2. *The Threshold:* When you call =setenv=, =libc= realizes the
   Kernel-provided stack space is fixed in size. It cannot simply "push"
   more data there without crashing into your program's actual data.
3. *The Relocation:* =libc= calls =realloc= or =malloc=. It copies all
   existing environment pointers to a new block in the *Heap*, adds your
   new variable, and updates the =environ= pointer in the =.data=
   segment to this new address.
4. *The "Stale" Pointer:* The =envp= variable inside =main= is just a
   local pointer on =main='s stack frame. It has no idea =libc= just
   moved the "source of truth." It still points to the old, original
   stack location.

** Summary of Why This Matters
:PROPERTIES:
:CUSTOM_ID: summary-of-why-this-matters
:END:
- *Library Developers:* If you write a library, you *must* use =environ=
  because you don't have access to =main='s arguments.
- *Security:* Sometimes attackers try to manipulate the stack. Knowing
  that =environ= can move to the heap is a key part of understanding
  process memory safety.
- *Consistency:* Always treat =envp= as a "read-only snapshot" of how
  the program started. Treat =environ= as the "current state."

* 8.4.6 Usign =fork= and =execve= to Run Programs
** Using =exit= in child process

Your "bad feeling" is actually a very sharp observation. You are
touching on a subtle but important distinction in Unix systems
programming regarding how processes terminate.

In the context of the CS:APP shell example, let's break down why
=exit(0)= vs =_exit(0)= matters on line 44.

--------------

*** The Short Answer: Why you are right
:PROPERTIES:
:CUSTOM_ID: the-short-answer-why-you-are-right
:END:
In a child process that has failed to =execve=, using *=_exit()=* is
generally safer and more "correct" than =exit()=.

- *=exit()=* is a high-level library call. It flushes standard I/O
  buffers and runs functions registered with =atexit()=.
- *=_exit()=* is a direct system call. It terminates the process
  immediately without any user-space cleanup.

*** The Problem with =exit()= in a Forked Child
:PROPERTIES:
:CUSTOM_ID: the-problem-with-exit-in-a-forked-child
:END:
When you =Fork()= (line 41), the child gets a *copy* of the parent's
memory, including the state of the I/O buffers (like =stdout=).

1. If the parent has data sitting in a buffer that hasn't been printed
   yet, that data is now in the child's buffer too.
2. If the child calls =exit()=, it will flush those buffers.
3. Later, when the parent eventually exits or flushes, it will flush the
   same data.
4. *Result:* You might see duplicate output in your terminal.

--------------

*** Why CS:APP uses =exit(0)= anyway
:PROPERTIES:
:CUSTOM_ID: why-csapp-uses-exit0-anyway
:END:
While =_exit()= is technically "cleaner" for child processes that fail
to exec, the authors of CS:APP often use =exit()= for a few reasons:

1. *Simplicity:* For students, =exit()= is familiar. Introducing
   =_exit()= requires explaining the difference between the C library
   and the system call interface prematurely.
2. *The =execve= behavior:* Since =execve= (line 42) wipes the process
   memory anyway if it /succeeds/, the only time line 44 is reached is
   if the command fails. In this simple shell, there usually isn't
   enough buffered I/O to cause the "duplicate output" bug.
3. *Standardization:* The wrapper =unix_error= and other CS:APP wrappers
   often use standard =exit=.

*** Logic of the Process Flow
:PROPERTIES:
:CUSTOM_ID: logic-of-the-process-flow
:END:
The flow of this specific code is:

- *Line 41:* Process splits.
- *Line 42:* Child tries to transform into a new program.
- *Success:* The child's memory (and I/O buffers) is destroyed and
  replaced.
- *Failure:* The child continues to line 44.

#+begin_quote
*Pro Tip:* If you ever write a production-grade shell or a daemon,
definitely use =_exit()= in the child path after a failed =exec=. It
prevents "double-flushing" and avoids triggering cleanup handlers that
were meant only for the parent.

#+end_quote

--------------

*** One more "Bad Feeling" you should have: Line 51

:PROPERTIES:
:CUSTOM_ID: one-more-bad-feeling-you-should-have-line-51
:END:
While you're looking at termination, look at line 51. The parent uses
=waitpid(pid, &status, 0)=. This shell is very basic---it doesn't handle
*zombie processes* for background jobs (=bg=). If you run 100 commands
with =&=, and the parent never "waits" for them, those 100 processes
will stay in the process table as zombies.

* 8.5.3 Receiving Signals

Write a program called snooze that takes a single command-line
argument, calls the snooze function from Problem 8.5 with this
argument, and then terminates. Write your program so that the user can
interrupt the snooze function by typing =Ctrl+C= at the keyboard. For
example:

#+BEGIN_SRC sh
linux> ./snooze 5
CTRL+C                      # User hits Crtl+C after 3 seconds
Slept for 3 of 5 secs.
linux>
#+END_SRC

#+BEGIN_SRC C :tangle ysnooze.c
  /* Practice Problem 8.7 (solution page 834) */
  /* See README.org */
  
  #include "../include/csapp.h"

  static unsigned int snooze(unsigned int secs) {
    unsigned int rc = sleep(secs);
    printf("Slept for %d of %d\n", secs - rc, secs);
    return rc;
  }

  static void handler(int sig) { return; }

  int main(int argc, char *argv[argc + 1]) {
    if (argc < 2 || strcmp(argv[1], "-h") == 0) {
      fprintf(stderr, "Usage: %s time_secs\n", argv[0]);
      exit(1);
    }

    if (signal(SIGINT, handler) == SIG_ERR) /* Install SIGINT handler */
      unix_error("signal() failed");

    (void)snooze(atoi(argv[1]));

    return EXIT_SUCCESS;
  }
#+END_SRC

The sleep() function returns prematurely whenever it receives a signal
that is not ignored. The default action in case of =SIGINT= is to
terminate the process, thus had we not caught the signal with our
one-statement handler, the process would be terminated right before
the =snooze()= call could ever assign any value to =rc= after
=sleep()= returns. The seemingly innocuous handler is fundamental in:

1. catching the signal;
2. preempting the default action;
3. allowing the control flow to return to the ASM statement after
   =call sleep=, then assigning a value to =rc= and reaching the
   visually important =ptrinf()= call.

On the terminal side check its default canonical mode of the =tty=
terminal with:

=$ stty -a=

The =echoctl= option is responsible for the echoing of control
characters like =^C= after pressing =Ctrl-C= keys. All control
characters echo representation come out of the scheme:

=^X = ASCII(X) & 0x1F=

Then to sent the =SIGINT= signal which the terminal assigns to =ASCII
0x03=:

='C' = 0x43=
=0x43 & 0x1F = 0x03=

You can toggle that control echo on and off with:

=$ stty [-]echoctl=

and run the 'snooze' process in between.

** Problem 8.20
Use execve to write a program called =myls= whose behavior is
identical to the =/bin/ls= program. Your program should accept the
same command-line arguments, interpret the identical environment
variables, and produce the identical output.

The =ls= program gets the width of the screen from the =COLUMNS=
environment variable. If =COLUMNS= is unset, then ls assumes that the
screen is 80 columns wide. 

#+BEGIN_SRC C :tangle ymyls.c
  /* Problem 8.20 */
  #include "../include/csapp.h" // IWYU pragma: keep

  int main(int argc, char *argv[], char *envp[]) {
    char *ls_path = "/bin/ls";

    if (execve(ls_path, argv, envp) == -1) {
      perror("execve() failed");
      exit(EXIT_FAILURE);
    }
  }
#+end_src

Using *=stty cols N=* to directly broadcast the terminal width is
actually the most reliable way to force =ls= (and therefore =myls=) to
respect a custom width, because it bypasses the environment variable
ambiguity and sets the terminal's reported size at the system level.

Here's why that works better than =COLUMNS= on modern GNU =ls=:

- *Terminal query wins:* GNU =ls= first asks the terminal for its width
  using =ioctl(TIOCGWINSZ)=. If it gets a valid answer, it ignores
  =COLUMNS=.
- *=COLUMNS= fallback:* Only if the terminal doesn't report a width
  (e.g., when output is redirected or piped) does =ls= fall back to the
  =COLUMNS= environment variable.
- *=stty cols N=:* This command changes the terminal's reported width to
  =N=. So when =ls= queries the terminal, it sees your forced width and
  formats accordingly.

*** Example
:PROPERTIES:
:CUSTOM_ID: example
:END:
#+BEGIN_SRC sh
stty cols 40
./myls   # output formatted to 40 columns

stty cols 80
./myls   # back to 80 columns
#+end_src

** Problem 8.22
Write your own version of the Unix system function

#+BEGIN_SRC C
  int mysystem(char *command);
#+end_src

The =mysystem= function executes command by invoking =/bin/sh -c
command=, and then returns after =command= has completed. If =command=
exits normally (by calling the =exit= function or executing a =return=
statement), then mysystem returns the command exit status. For
example, if command terminates by calling =exit(8)=, then =mysystem=
returns the value =8=. Otherwise, if =command= terminates abnormally,
then =mysystem= returns the status returned by the shell.

*** The purpose of =system= library function 
Let's place =system()= in the context of the classic *process‑spawning
chain* (=fork=, =exec*=, =wait*=).

**** What =system()= Does
:PROPERTIES:
:CUSTOM_ID: what-system-does
:END:
- =system(const char *command)= is a *library function* in C (from
  =<stdlib.h>=).
- It takes a shell command string (e.g., ="ls -l /tmp"=) and runs it as
  if you typed it in your shell.
- Internally, it does the following:
  1. *Forks* a child process.
  2. In the child, it *execs* the user's shell (usually =/bin/sh=) with
     the command string.
  3. The parent process *waits* for the child to finish, then returns
     the child's exit status.

So =system()= is essentially a convenience wrapper around the =fork= →
=exec= → =wait= chain.

**** Comparison
:PROPERTIES:
:CUSTOM_ID: comparison
:END:
| Aspect                 | Manual =fork= + =exec= + =wait=          | =system()=                                                           |
|------------------------+------------------------------------------+----------------------------------------------------------------------|
| Control over arguments | Full control (you pass argv, envp)       | Limited --- you pass a single command string                         |
| Shell involvement      | Not required (execs program directly)    | Always runs through a shell (=/bin/sh -c=)                           |
| Error handling         | You handle =fork=/=exec= errors yourself | =system()= returns =-1= on fork error, or child's exit status        |
| Convenience            | Verbose, but precise                     | Very easy, one call                                                  |
| Security               | Safer (no shell parsing)                 | Risky if command string comes from untrusted input (shell injection) |

**** Example
:PROPERTIES:
:CUSTOM_ID: example
:END:
Manual approach:

#+BEGIN_SRC C
  pid_t pid = fork();
  if (pid == 0) {
      execl("/bin/ls", "ls", "-l", "/tmp", (char *)NULL);
      perror("execl failed");
      exit(EXIT_FAILURE);
  } else {
      int status;
      waitpid(pid, &status, 0);
  }
#+END_SRC

Using =system()=:

#+BEGIN_SRC C
  system("ls -l /tmp");
#+END_SRC

**** Purpose in the Chain
:PROPERTIES:
:CUSTOM_ID: purpose-in-the-chain
:END:
- =system()= is a *high‑level shortcut*: it hides the details of =fork=,
  =exec=, and =wait=.
- It's great for quick commands, scripts, or prototyping.
- But if you need fine‑grained control over arguments, environment, or
  file descriptors, you drop down to the explicit =fork=/=exec=/=wait=
  chain.

--------------

So in short: *=system()= is a convenience wrapper that spawns a shell to
run a command, internally relying on =fork=, =exec=, and =wait=.*

*** =mysystem=
#+BEGIN_SRC C :tangle ysystem.c
  /* Problem 8.22 */
  #include "../include/csapp.h" /* IWYU pragma: keep */

  static int mysystem(char *command) {
    pid_t pid;
    int status;
    const char *shell_path;

    shell_path = getenv("SHELL");
    if (shell_path == NULL)
      shell_path = "/bin/sh";

    if (command == NULL)
      return (access(shell_path, X_OK) == 0) ? 1 : 0;

    if ((pid = fork()) < 0)
      return -1; /* errno set by fork */

    if (pid == 0) {
      execl(shell_path, shell_path, "-c", command, (char *)NULL);
      _exit(127); /* exec failed */
    }

    if (waitpid(pid, &status, 0) < 0)
      return -1; /* errno set by waitpid */

    if (WIFEXITED(status)) {
      return WEXITSTATUS(status);
    } else {
      return status; /* abnormal termination */
    }
  }

  int main() {
    int status = mysystem("ls -l /tmp");
    if (status == -1) {
      fprintf(stderr, "ERROR: mysystem(): %s\n", strerror(errno));
      exit(EXIT_FAILURE);
    }

    printf("\nCommand exit status: %d\n", status);
    exit(EXIT_SUCCESS);
  }
#+END_SRC

** Problem 8.23

Although 5 signals are sent from child to parent, at parent's exit
src_C{counter = 2;}.

The raison is signals are not queued. After installing a =s_handler=
to catch =s= signal, a first =s= signal is caught and handles by
=s_handler=. If a second =s= signal arrives while =s_handler= is
running, the second signal is added to the pending vector by the
system for later handling. Any subsequent arrival of =s= type signal
while the handler is running are not queued but outright discarded and
lost for good. When the handler returns the system sees in the pending
vector the "ON" bit for the =s= signal and forces the process to
handle the pending signal setting =counter= to 2.

#+BEGIN_SRC C :tangle counterprob.c
  /* Problem 8.23 */

  #include "../include/csapp.h"

  int counter = 0;

  void sigint_handler(int sig) {
    counter++;
    sleep(1); /* Do some work in the handler */
    return;
  }

  int main() {
    int i;

    Signal(SIGUSR2, sigint_handler);

    if (Fork() == 0) { /* Child */
      for (i = 0; i < 5; i++) {
	Kill(getppid(), SIGUSR2);
	printf("sent SIGUSR2 N°%d to parent\n", i + 1);
      }
      exit(0);
    }

    Wait(NULL);
    printf("counter=%d\n", counter);
    exit(0);
  }
#+end_src

** Problem 8.24
Modify the program in Figure 8.18 so that the following two conditions
are met:
1. Each child terminates abnormally after attempting to write to a
   location in the read-only text segment.
2. The parent prints output that is identical (except for the PIDs) to
   the following:
     - =child 12255 terminated by signal 11: Segmentation fault=
     - =child 12254 terminated by signal 11: Segmentation fault=

Hint: Read the man page for psignal(3).

#+BEGIN_SRC C :tangle segfault.c
  /* Problem 8.24 */
  #include "../include/csapp.h"
  #define N 2

  /* Dummy function on the .text segment */
  void foo() {};

  int main() {
    size_t i;
    pid_t pid;
    int status;
    int *p = (int *)foo; /* Retreive address on the read-only .text segment */

    /* Parent create N children */
    for (i = 0; i < N; i++)
      if (Fork() == 0) {
	,*p = 1; /* Overwrite on .text segment: trigger segmentation fault */
	exit(100 + i); /* Control flow never gets here */
      }

    /* Parent reaps N children in no particular order */
    while ((pid = waitpid(-1, &status, 0)) > 0) {
      if (WIFEXITED(status)) {
	printf("[PID:%ld]child process: terminated normally: status = %d\n",
	       (long)pid, WEXITSTATUS(status));
      } else if (WIFSIGNALED(status)) {
	int sig = WTERMSIG(status);
	char s[MAXBUF];
	snprintf(s, MAXBUF, "child %ld terminated by signal %d", (long)pid, sig);
	psignal(sig, s);
      }
    }

    /* The only normal termination is if there are no more children */
    if (errno != ECHILD)
      unix_error("waitpid() failed");

    exit(0);
  }
#+end_src
