** 1.2 Pros and Cons of Processes

*** Practice Problem 12.1
[[file:echoserverp.c][Figure 12.5 echoserverp]] demonstrates a concurrent server in which the
parent process creates a child process to handle each new connection
request. Trace the value of the reference counter for the associated
file table for Figure 12.5 (solution page 1072).

**** Answer
:PROPERTIES:
:VISIBILITY: folded
:END:
When the parent process on the concurrent server starts executing, the
reference counter to =connfd= increments from 0 to 1 for the
associated file table. When this parent process forks the child
process, the reference counter is incremented from 1 to 2. When the
parent closes its copy of the descriptor, the reference count is
decremented from 2 to 1. Similarly, when the child’s end of connection
closes, the reference counter is decremented from 1 to 0.

*** Practice Problem 12.2

connected descriptor of the parent process, the code would still be

(solution page 1072).

#+begin_quote

**** Answer
:PROPERTIES:
:VISIBILITY: folded

When a process terminates for any reason, the kernel closes all open
descriptors. Thus, the parent’s copy of the connected file descriptor
will be closed automatically when the parent exits.

*** The State Machine Philosophy
The state machine framing is one of those conceptual jumps that feels
heavier than it should because CS:APP introduces it somewhat abruptly
without much scaffolding.

Let me try to ground it simply.

In the forked server, each client gets its own process with its own
stack, its own local variables, its own execution flow.  The OS handles
the bookkeeping of "where am I in serving client X" implicitly through
the process context.

In the event-driven multiplexed server, you have *one process, one
flow of control*.  So when you're in the middle of serving client X
and =select= tells you client Y has data ready, you have to *remember
where you were* with client X yourself.  The state machine is just
that memory — the =pool= struct is encoding "for each connected
client, what state is their conversation in right now."

Each client is a little machine that can be in states like: waiting
for data, reading data, echoing back.  The =select= loop is the engine
that looks at all the machines, finds the ones that have something to
do, advances them one step, then loops.

The reason it feels heavy is that with processes or threads, *the call
stack IS the state*.  You just block on =read= and the OS preserves
everything.  With I/O multiplexing you can never block, so you have to
make that implicit call stack state *explicit* in a data structure.

That tension — implicit state via blocking vs explicit state via
structs — is really the philosophical heart of the whole chapter.

*** Cardinality and Scalability with =select()=

The first argument to =select()= define as =pool.ready_set + 1= is
arithmetic trick to map 0-based index to 1-based count relying on the
fact that file descriptor are non-negative integer and the highest is
the bound of =select()= range.

A practical consequence worth keeping in mind: This is also why
=select()= has scalability problems.  It has to scan the entire range
=[0, nfds-1]= every call, even if a handful of those fds are actually
set in your fd_set.  If you have nothing else but one fd with value
1023, it still scans 1042 slots.

That is part of why =epoll= was invented.  It works on an event list
rather than a dense bitmap scan, so it scales with the number of
*active* fds rather than the magnitude of the largest fd value (Ref:
TLPI.63 covers that contrast in satisfying detail).

*** Pedagogical tradeoffs in echoservers design

In the =echoservers.c= design the declaration of =pool= struct
produces a massive stack allocation of 1024 slots worth in total
*8MiB*, upfront even when the server services only one client.

That is unpractical and indefensible in production.  CS:APP is
essentially showing the concept with training wheels on.  The pool
struct makes the /state machine/ idea *visible and tangible,* which is
worth the ugliness for learning purposes.  But Kerrisk in TLPI.63 will
show you what a more serious design looks like.

*** The crucial observation CS:APP is building toward

Notice that =cnt++= compiles to *three separate instructions*.

- *L* --- Load =cnt= from memory into a register
- *U* --- Update (increment) the register
- *S* --- Store the register back to memory

These three are not atomic. The scheduler can interrupt thread i
between any of them and hand control to thread j. Thread j then does
its own L/U/S on the stale value it loaded, and when thread i resumes
and does its S it *overwrites thread j's work*.

That is the race condition. The assembly makes it viscerally visible
in a way the C source completely hides. =cnt++= looks like one thing
but is three things, and that gap between H and S is where concurrency
goes wrong.
